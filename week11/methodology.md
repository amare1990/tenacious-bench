# Methodology — Tenacious-Bench v0.1

## 1. Path Declaration

I choose **Path B: preference-tuned judge / critic**.

## 2. Why Path B

Week 10 evidence shows outputs that are fluent but unsafe for Tenacious: weak grounding, generic tone, internal bench leakage, unsupported capacity claims, pricing-policy violations, and condescending AI-maturity framing. These are accept/reject failures. The intervention is therefore a trained critic that gates or re-ranks outbound candidates before delivery.

See `methodology_rationale.md` for trace-level rationale and paper references.

## 4. Four-Mode Dataset Authoring

Target distribution is documented as approximately **30% trace-derived / 30% programmatic / 25% multi-LLM synthesis / 15% hand-authored adversarial**. The current dataset audit is 215 tasks: 27 trace-derived, 143 programmatic, and 45 synthesis. Hand-authored adversarial generation is implemented as a committed authoring mode and can be merged into the next dataset revision.

Source modes:

1. `trace_derived`: mined from Week 10 traces and known probe failures.
2. `programmatic`: generated from structured slots: company size, segment, headcount, stack, bench state, and AI-maturity score.
3. `multi_llm_synthesis`: generated by separated model roles.
4. `hand_authored_adversarial`: manually designed edge cases for subtle policy violations.

Every task carries `source_mode` and `metadata.authoring_mode` so provenance remains inspectable.

## 5. Multi-LLM Routing and Anti-Leakage

Model roles are separated in `week11/generation_scripts/multi_llm_routing_policy.json`:

- frontier seed author: GPT-4.1 / Claude 3.7 family
- dev-tier bulk generator: Qwen / Llama family
- judge filter: Gemini / Mistral family

Rules:

- the same model must never generate and judge the same task;
- the judge family must differ from the synthesis source family;
- filter logs record generator model, judge model, pass/fail, and reasons;
- the rationale cites Li et al. (2025) to reduce preference leakage.

## 6. Judge Filter Pipeline

`week11/generation_scripts/judge_filter_pipeline.py` scores each task on four pointwise dimensions: input coherence, ground-truth verifiability, rubric clarity, and Tenacious policy relevance. Inclusion requires all dimensions >= 3 and mean score >= 3.5. It also performs pairwise near-duplicate checks. Dev-tier filtering handles the full batch; eval-tier calibration is reserved for a 50-task sampled subset.

## 7. Contamination Prevention

`week11/generation_scripts/contamination_check.py` performs:

- sub-8-gram overlap checks across held-out/train and held-out/dev;
- cosine similarity checks using a cheap hashing embedding proxy with threshold 0.84;
- time-shift verification for tasks that reference public signals;
- structured JSON reporting.

## 8. Training

`week11/training/train_path_b_judge.py` uses TRL `DPOTrainer` for Path B, LoRA-only fine-tuning, fixed seed, explicit learning rate, batch size, LoRA rank/alpha, epochs, warmup, scheduler, and loss logging.

## 9. Ablations

`week11/ablations/evaluate_path_b_heldout.py` compares:

- Delta A: trained critic intervention vs Week 10 baseline;
- Delta B: same backbone with prompt-only critic vs trained critic;
- Delta C: τ²-Bench reference as informational contrast, not a rerun;
- cost-Pareto instrumentation: timing, token estimates, and per-task cost fields.
